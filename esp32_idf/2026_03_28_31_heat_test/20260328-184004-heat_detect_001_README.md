# MAX31850KATB+ Temperature Sensor Driver Troubleshooting Report

**Date**: 2026-03-28  
**Branch**: `feature/heating`  
**Task Description**: Troubleshoot and fix 1-Wire communication timing issues for stable temperature reading from MAX31850 sensors on ESP32-S3 GPIO14

---

## 1. Background

### Hardware Configuration
- **MCU**: ESP32-S3-DevKitC-1 (240MHz, dual-core)
- **Sensors**: 4× MAX31850KATB+ (K-type thermocouple digitizer with cold-junction compensation)
- **Bus Protocol**: 1-Wire, 4 sensors paralleled on single bus
- **GPIO**: IO14 (GPIO_NUM_14)
- **Pull-ups**: 4.7KΩ per sensor (independent pull-up to 3.3V)
- **Power**: 3.3V external supply (non-parasitic mode)

### Initial Implementation Status
The initial driver implementation (Phase 1) included:
- GPIO14 initialization with open-drain mode
- 1-Wire ROM search algorithm for auto-enumeration
- Temperature reading with CRC validation
- Non-blocking polling task with state machine
- Fault detection (open circuit, short to GND/VCC)

### Issue Description
After initial implementation, the driver exhibited the following symptoms:
- **First read success**: Initial temperature reading (1.58°C) was correct
- **Subsequent reads failed**: All following reads produced various errors
- **Error types**: CRC errors, Short to GND, Open circuit, temperature out of range (-640°C, -576°C)
- **Byte shift observed**: Raw scratchpad showed `[D8 00 ...]` instead of `[00 D8 ...]`

## 2. Problem Analysis

### 2.1 Log Analysis

#### Symptom 1: Byte Order Corruption
```
Raw scratchpad: [DC 00 50 14 F0 FF FF FF EA]
                ↑↑ ↑↑
Expected:      [00 DC ...]  (Temperature LSB first, MSB second)
Actual:        [DC 00 ...]  (Bytes swapped)
```

#### Symptom 2: CRC Errors After Successful ROM Search
```
Found device 1: ROM ID 3B00000FE1C5A872
Found device 2: ROM ID 3B08000FE2889E46
Found device 3: ROM ID 3B00000FE1C5A872  ← Duplicate ID (wrong!)
Search error at bit 59: 1/1
```

#### Symptom 3: Data Instability Across Retries
```
Attempt 1: [E8 08 B0 14 F2 ...]  ← CRC failed
Attempt 2: [E9 00 B0 14 F2 ...]  ← CRC failed
Success:   [E9 00 B0 14 F2 ...]  ← CRC passed but data incorrect
```

### 2.2 Root Cause Hypothesis

| Hypothesis | Evidence | Test Result |
|------------|----------|-------------|
| Mutex interference during conversion | Mutex released during 150ms conversion wait | Fixed - no improvement |
| Interrupt/task switch during bit timing | Critical section added | Fixed - no improvement |
| Insufficient timing margins | Increased delays 2x-3x | Fixed - no improvement |
| Bus capacitance/risetime | ROM search also fails occasionally | **Likely cause** |
| Hardware/wiring issue | First read always succeeds | **Under investigation** |

### 2.3 Key Observations

1. **First read always succeeds** - Suggests initial state is correct
2. **ROM search occasionally finds duplicate IDs** - Bit-level corruption during search
3. **Byte shift pattern consistent** - Always `[XX 00]` instead of `[00 XX]`
4. **Conservative timing (2-3x slower) has no effect** - Rules out simple timing issues

## 3. Fix Attempts

### Attempt 1: Fix Mutex Timing (Commit: c9d7b07)

**Theory**: Other tasks accessing GPIO14 during conversion wait period

**Changes**:
```c
// Before: Release mutex during conversion
max31850_start_conversion(rom_id);
xSemaphoreGive(s_onewire_mutex);  // ← Release here
vTaskDelay(pdMS_TO_TICKS(150));
xSemaphoreTake(s_onewire_mutex, ...);

// After: Keep mutex locked throughout
max31850_start_conversion(rom_id);
// Keep mutex locked
vTaskDelay(pdMS_TO_TICKS(MAX31850_CONVERSION_TIME_MS));
max31850_read_scratchpad(rom_id, scratchpad);
xSemaphoreGive(s_onewire_mutex);
```

**Result**: ❌ No improvement

for (int retry = 0; retry < 3; retry++) {

### Attempt 4: Critical Section Protection (Commit: f9a22ee)

**Theory**: FreeRTOS task switching corrupting microsecond timing

**Changes**:
```c
static portMUX_TYPE s_onewire_mux = portMUX_INITIALIZER_UNLOCKED;

static esp_err_t onewire_read_bit(uint8_t *bit)
{
    portENTER_CRITICAL(&s_onewire_mux);  // Disable interrupts
    
    // Critical timing sequence
    gpio_set_direction(pin, GPIO_MODE_INPUT_OUTPUT_OD);
    gpio_set_level(pin, 0);
    esp_rom_delay_us(1);
    gpio_set_direction(pin, GPIO_MODE_INPUT);
    esp_rom_delay_us(13);
    *bit = gpio_get_level(pin);
    
    portEXIT_CRITICAL(&s_onewire_mux);   // Re-enable interrupts
}
```

**Applied to**: `onewire_reset()`, `onewire_write_bit()`, `onewire_read_bit()`

**Result**: ❌ No improvement

## 8. Conclusions

### Software Fixes Exhausted
All software-level fixes have been attempted without success:
1. ✅ Mutex protection throughout operation
2. ✅ Critical section for microsecond timing
3. ✅ Conservative timing (2-3x slower than standard)
4. ✅ Read retry with CRC validation
5. ✅ Bit-level and byte-level timing delays

### Likely Hardware Root Cause
Given that:
- First read always succeeds
- ROM search occasionally fails (before any temperature conversion)
- Byte shift pattern is consistent
- Conservative timing has no effect

**Most likely causes** (in order of probability):
1. **Bus capacitance** - 4 sensors in parallel with 4.7KΩ pull-ups insufficient
2. **Signal integrity** - Wiring/layout issues on GPIO14
3. **GPIO14 hardware limitation** - Drive strength insufficient for multi-device bus
4. **Sensor hardware defect** - One or more sensors corrupting bus
