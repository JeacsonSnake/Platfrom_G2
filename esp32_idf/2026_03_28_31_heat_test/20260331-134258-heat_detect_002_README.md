## 1. Background

### Historical Issues
Based on previous development records (2026-03-27 to 2026-03-28), the MAX31850 driver exhibited persistent timing issues:
- **First read success, subsequent reads failed**: Initial temperature reading was correct, but all following reads produced errors
- **Various error types**: CRC errors, Short to GND, Open circuit, temperature out of range (-640°C, -576°C)
- **Byte shift pattern**: Raw scratchpad showed `[XX 00 ...]` instead of `[00 XX ...]`
- **Root cause**: Previous software bit-bang implementation had microsecond-level timing instability

### Previous Attempts (Exhausted)
All software fixes had been attempted without success:
- Mutex protection throughout operation
- Critical section for microsecond timing
- Conservative timing (2-3x slower than standard)
- Read retry with CRC validation
- Bit-level and byte-level timing delays

### Hardware Configuration
- **MCU**: ESP32-S3-DevKitC-1 (240MHz, dual-core)
- **Sensors**: 4× MAX31850KATB+ (K-type thermocouple digitizer)
- **Bus Protocol**: 1-Wire, 4 sensors paralleled on single bus
- **GPIO**: IO14 (GPIO_NUM_14)
- **Pull-ups**: 4.7KΩ per sensor (independent pull-up to 3.3V)
- **Power**: 3.3V external supply (non-parasitic mode)

## 2. Driver Reimplementation Design

### 2.1 Requirements

| Requirement | Description |
|-------------|-------------|
| Protocol | 1-Wire with precise timing (GPIO bit-bang + critical section) |
| Device Discovery | Auto-enumerate 4 sensors at initialization via Search ROM |
| Access Method | Match ROM (0x55) + 64-bit ROM ID for individual sensor access |
| Polling Rate | 1-second interval for all 4 sensors (non-blocking state machine) |
| Conversion Time | ≥100ms wait between Convert T and Read Scratchpad |
| CRC Validation | X8+X5+X4+1 polynomial verification on scratchpad data |
| Fault Detection | Open circuit, Short-GND, Short-VCC detection |
| Error Recovery | Mark offline after 3 consecutive failures, periodic retry |

### 2.2 State Machine Design

```
POLL_STATE_IDLE ──→ POLL_STATE_START_CONVERT ──→ POLL_STATE_WAIT_CONVERSION
       ↑                                              │
       └──────────────────────────────────────────────┘
       │
       └──→ POLL_STATE_READ_DATA ──→ POLL_STATE_NEXT_SENSOR
```

**State Descriptions**:
- `START_CONVERT`: Send Convert T (0x44) command to selected sensor
- `WAIT_CONVERSION`: Delay 100ms (non-blocking via vTaskDelay)
- `READ_DATA`: Send Read Scratchpad (0xBE), receive 9 bytes
- `NEXT_SENSOR`: Move to next sensor or restart cycle

### 2.3 Key Data Structures

```c
// Error codes
typedef enum {
    MAX31850_OK = 0,            // Success
    MAX31850_ERR_OPEN,          // Thermocouple open circuit
    MAX31850_ERR_SHORT_GND,     // Short to GND
    MAX31850_ERR_SHORT_VCC,     // Short to VCC
    MAX31850_ERR_CRC,           // CRC validation failed
    MAX31850_ERR_TIMEOUT,       // Communication timeout
    MAX31850_ERR_OFFLINE,       // Sensor marked offline
    MAX31850_ERR_INVALID_ID,    // Invalid sensor ID
    MAX31850_ERR_BUS_FAULT,     // Bus electrical fault
    MAX31850_ERR_NOT_FOUND,     // Device not found
} max31850_err_t;

// Sensor status structure
typedef struct {
    uint8_t rom_id[8];          // 64-bit ROM ID (family + serial + CRC)
    float temperature;          // Last read temperature (°C)
    int16_t raw_temp;           // Raw 16-bit temperature value
    uint8_t fault_reg;          // Fault register (byte 4 of scratchpad)
    max31850_err_t last_error;  // Last error code
    uint8_t fail_count;         // Consecutive failure counter
    bool online;                // Online status
    bool data_valid;            // Data validity flag
    uint32_t last_read_time;    // Last successful read timestamp
} max31850_sensor_t;
```

## 6. Current Status and Findings

### 6.1 Working Features
- ✅ GPIO14 initialization with open-drain mode
- ✅ GPIO diagnostic (5 tests pass)
- ✅ Reset/Presence detection (device responds)
- ⚠️ ROM search (intermittent - occasionally finds duplicates)
- ⚠️ Skip ROM read (returns data but CRC fails)

### 6.2 Key Findings

**Finding 1: GPIO Hardware is Functional**
- GPIO14 open-drain mode works correctly
- Pull-up resistor functioning (3.3V idle)
- Device responds to Reset pulse (Presence detected)

**Finding 2: Basic 1-Wire Communication Works**
- Skip ROM command successfully reads 9 bytes
- Data format appears correct (Family Code 0x3B visible)
- Temperature conversion and scratchpad read execute

**Finding 3: Data Integrity Issues**
- ROM Search consistently fails CRC validation
- Skip ROM read returns data but CRC check fails
- Pattern suggests bit-level sampling errors

**Finding 4: Data Pattern Analysis**
```
Skip ROM Scratchpad: [00 00 00 18 F0 FF FF FF 10]
Expected:           [00 00 00 18 F0 00 00 00 XX] (XX = valid CRC)
Actual Bytes 5-7:   FF FF FF (should be 00 00 00)
```
- Bytes 5-7 consistently reading as 0xFF instead of 0x00
- Suggests sampling too early or bus not stable

### 6.3 Root Cause Hypothesis

Given that:
- Reset/Presence works correctly
- Skip ROM can read data (but with errors)
- CRC fails consistently
- Conservative timing has limited effect

**Most Likely Causes** (in order of probability):
1. **Read sampling timing**: Sample point may still be slightly off
2. **Bus rise time**: 4.7KΩ pull-up may be too slow for multi-device bus
3. **GPIO drive strength**: ESP32-S3 may need stronger pull-up assistance
4. **Signal integrity**: Wiring/layout issues on GPIO14

## 9. Troubleshooting Guide

### Issue 1: "No MAX31850 devices found"

**Check GPIO diagnostic output**:
- If Test 1 fails (shows 0): Check 4.7KΩ pull-up resistor connection
- If Test 2 fails (shows 1): Check GPIO14 short to VCC
- If Test 3 fails (shows 0): Check bus stuck low (short to GND)

**Check Reset/Presence**:
```
Reset: presence_level=0, presence=YES   ← Good
Reset: presence_level=1, presence=NO    ← No device responding
```

### Issue 2: "ROM ID CRC error"

**Check ROM data pattern**:
```
ROM data: 3B-XX-XX-XX-XX-XX-XX-XX       ← Family code 0x3B is correct
ROM data: 00-XX-XX-XX-XX-XX-XX-XX       ← No device response
ROM data: FF-FF-FF-FF-FF-FF-FF-FF       ← Bus stuck high
```

**If Skip ROM succeeds but Search ROM fails**:
- Issue likely in Search ROM bit handling logic
- Consider using single sensor with Skip ROM as workaround

### Issue 3: "CRC Error" on Skip ROM

**Check scratchpad data**:
```
Scratchpad: [00 00 00 18 F0 FF FF FF 10]  ← Bytes 5-7 are FF (wrong)
Expected:   [00 00 00 18 F0 00 00 00 XX]  ← Should be 00
```

**Solutions to try**:
1. Reduce pull-up resistor to 2.2KΩ or 1KΩ for faster rise time
2. Increase read sample delay further
3. Check for signal integrity issues with oscilloscope

Disconnect 3 of 4 sensors, test with only 1:
```
If single sensor works → Bus capacitance/interference issue

Measure 1-Wire signal on GPIO14:
```
Check: Rise time (<1μs required)
