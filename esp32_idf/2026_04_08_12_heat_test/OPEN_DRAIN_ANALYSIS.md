### 1.1 差异摘要


**仅修改了读时序参数**，开漏测试函数 `gpio_test_open_drain()` **完全未改动**。

### 1.2 开漏测试代码（完全相同）

```c
static bool gpio_test_open_drain(gpio_num_t gpio)
{
    // 测试1: 拉低总线
    gpio_set_level(gpio, 0);
    esp_rom_delay_us(100);          // 固定100us延时
    int level_low = gpio_get_level(gpio);
    
    // 测试2: 释放总线
    gpio_set_level(gpio, 1);
    esp_rom_delay_us(100);          // 固定100us延时
    int level_high = gpio_get_level(gpio);
    
    return (level_low == 0) && (level_high == 1);
}
```

**关键发现**: 
- 开漏测试使用 **固定的100us延时**
- 与 `ONE_WIRE_READ_*` 参数 **完全无关**
- 两次提交的开漏测试代码 **完全一致**

**同一版本代码，不同时间测试结果不同！**

### 2.2 结论

开漏测试的不一致 **不是软件问题**，而是 **硬件问题**：

1. **上拉电阻接触不良** - 4.7KΩ电阻可能存在虚焊或接触问题
2. **电源波动** - 3.3V电源可能不稳定
3. **GPIO引脚问题** - GPIO14引脚可能存在物理损坏
4. **总线电容** - 4个传感器并联导致总线电容过大

## 3. 数据手册参考

### 3.1 上拉电阻要求

根据 `max31850-max31851.pdf` **Page 15**:

> "The 1-Wire bus requires an external pullup resistor of **approximately 5kΩ**"

当前使用 **4.7KΩ**，在规格范围内。

### 3.2 问题分析

虽然4.7KΩ在规格内，但对于 **4个设备并联** 的情况：
- 总线电容 ≈ 4 × 单设备电容
- 上升时间 ≈ 2.2 × R × C
- 4.7KΩ可能太慢，导致开漏测试在100us内无法上升到高电平

**建议**: 更换为 **2.2KΩ** 或 **1KΩ** 上拉电阻。

## 4. 软件改进方案

虽然根本原因是硬件，但可以通过软件增加开漏测试的稳定性：

### 4.1 增加等待时间

```c
// 原代码: 100us等待
esp_rom_delay_us(100);

// 建议: 增加到200us或更长
esp_rom_delay_us(200);
```

### 4.2 多次采样确认

```c
// 多次采样，确保稳定性
int high_count = 0;
for (int i = 0; i < 5; i++) {
    esp_rom_delay_us(50);
    if (gpio_get_level(gpio) == 1) high_count++;
}
bool test_pass = (high_count >= 3);  // 多数表决
```

### 4.3 放宽开漏测试要求

如果开漏测试失败但其他功能正常，可以改为警告而非错误：

```c
if (!gpio_test_open_drain(gpio_num)) {
    ESP_LOGW(TAG, "GPIO open-drain test FAILED, but continuing...");
    // 不返回错误，继续初始化
}
```

## 5. 推荐解决方案

### 短期（软件 workaround）

1. **增加开漏测试等待时间** - 100us → 200us
2. **实施方案B时序** - 优化读时序参数
3. **放宽测试要求** - 开漏测试失败继续运行

### 长期（硬件修复）

1. **更换上拉电阻** - 4.7KΩ → 2.2KΩ
2. **检查焊接** - 确保R1焊接良好
3. **检查GPIO14** - 确保引脚无损坏

---

## 6. 与读时序的关系

虽然开漏测试不一致不是软件造成的，但**稳定的开漏测试是正确读时序的前提**：

- 开漏测试失败 → 上拉电阻无法将总线拉高
- 读时序中主机释放总线后 → 从设备无法正确驱动
- 结果 → 采样到错误电平，位反转，CRC失败

**因此必须先解决开漏测试问题，才能验证读时序参数！**

- 测试结果不一致是**硬件问题**
- 需要先通过软件/硬件手段**稳定开漏测试**
- 然后才能验证读时序参数
