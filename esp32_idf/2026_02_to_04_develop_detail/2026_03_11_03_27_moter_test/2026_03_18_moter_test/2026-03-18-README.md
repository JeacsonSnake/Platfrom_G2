# ESP32S3 Motor Control Soft-Start Fix and PWM Initialization Fix

**日期**: 2026-03-18  
**分支**: `feature/motor-control-config`  
**任务描述**: 修复PWM初始化时电机意外转动问题，验证软启动重置逻辑有效性，分析MQTT消息延迟根因

---

## 1. 背景

### 历史问题回顾
根据 2026-03-16 的测试记录，ESP32S3 电机控制存在以下问题：
- **PWM初始化问题**: 启动时电机意外转动，初始duty为0（对于反相逻辑意味着全速）
- **软启动失效**: 部分指令显示 `startup=11` 但未触发reset，导致启动电流过大
- **MQTT消息延迟**: PCNT_EVENT反馈延迟6秒以上，命令执行不及时
- **网络不稳定**: TCP_TRANSPORT_ERROR频繁，连接保持率低

### 本次任务目标
1. 修复PWM初始化duty值，防止启动时电机转动
2. 验证并修复软启动reset逻辑，确保0→非零速度转换时正确触发
3. 分析MQTT消息延迟根因，区分代码问题与环境问题

## 2. 问题分析与修复方案

### 2.1 PWM初始化Duty修复

#### 问题分析
| 项目 | 问题描述 |
|------|----------|
| 硬件特性 | CHB-BLDC2418使用反相PWM逻辑：High=OFF, Low=ON |
| 错误配置 | `LEDC_DUTY` 初始化为0，对应电机全速转动 |
| 影响 | 每次ESP32启动或复位时，电机会突然转动 |

#### 修复方案
将 `LEDC_DUTY` 从 `0` 改为 `8191`（13-bit分辨率下的最大值）：
```c
// main/main.h
#define LEDC_DUTY       (8191)      // 0=OFF (inverted logic), was 0
```

### 2.2 软启动Reset逻辑验证

#### 问题分析
| 现象 | 原因 |
|------|------|
| `startup=11` 无reset | `prev_target_speed[]` 数组未正确跟踪速度变化 |
| 启动电流过大 | 软启动未触发，PID直接输出大占空比 |
| 59%速度超调 | 无软启动限制时，实际速度405 vs 目标255 |

#### 修复方案
在 `main/pid.c` 中添加0→非零速度转换检测：
```c
// 检测电机启动转换（0→非零）
static double prev_target_speed[4] = {0};
if (temp > 0 && prev_target_speed[index] == 0) {
    if (!startup_phase || startup_counter > 10) {
        startup_phase = 1;
        startup_counter = 0;
        ESP_LOGI(TAG, "Motor %d soft-start reset", index);
    }
}
prev_target_speed[index] = temp;
```

### 2.3 MQTT延迟问题分析

#### 问题分类
| 问题 | 类型 | 根因 |
|------|------|------|
| PCNT_EVENT缺失 | 网络层 | MQTT断线导致消息丢失 |
| 6秒延迟 | 网络层 | TCP重连耗时 |
| 命令丢失 | 网络层 | QoS 1缓冲区溢出 |

#### 分析结论
- **非代码问题**: 电机控制逻辑、PID算法、PWM输出均正常
- **网络环境问题**: WiFi信号-70dBm，MQTT连接保持率15.38%~50%
- **建议**: 改善WiFi信号，添加本地命令缓冲

## 3. 实现方案

### 3.1 修改文件

#### `main/main.h`
```c
// PWM配置修正
#define LEDC_DUTY       (8191)      // Was 0, now 8191 (OFF state for inverted logic)
#define LEDC_FREQ       (20000)     // 20KHz per CHB-BLDC2418 spec
```

#### `main/pid.c`
```c
// PID计算函数中添加软启动reset检测
double PID_Calculate(int index, double setpoint, double actual)
{
    // ... existing code ...
    
    // 软启动状态变量（静态，保持状态）
    static int startup_phase = 1;
    static int startup_counter = 0;
    static double max_pwm_during_startup = 3000;
    
    // 检测电机启动转换（0→非零）
    static double prev_target_speed[4] = {0};
    if (temp > 0 && prev_target_speed[index] == 0) {
        if (!startup_phase || startup_counter > 10) {
            startup_phase = 1;
            startup_counter = 0;
            ESP_LOGI(TAG, "Motor %d soft-start reset", index);
        }
    }
    prev_target_speed[index] = temp;
    
    // 应用软启动限制
    if (startup_phase && startup_counter <= 10) {
        double progress = startup_counter / 10.0;
        double current_max = max_pwm_during_startup + (8191 - max_pwm_during_startup) * progress;
        if (new_input > current_max) new_input = current_max;
        startup_counter++;
    } else {
        startup_phase = 0;
    }
    
    // ... rest of code ...
}
```

#### `main/pwm.c`
验证初始化日志输出：
```c
void pwm_init() {
    // ... timer config ...
    for(int i = 0; i < 4; i++) {
        ledc_channel_config_t ledc_channel = {
            // ... other configs ...
            .duty = LEDC_DUTY,  // 8191 = OFF (fixed)
            // ...
        };
        ledc_channel_config(&ledc_channel);
        ESP_LOGI(TAG, "PWM channel %d initiated on GPIO%d with initial duty %d (inverted logic).", 
                 i, pwm_gpios[i], LEDC_DUTY);
    }
}
```

## 7. 问题解决记录

### 问题：电机启动时意外转动
**现象**: ESP32启动或复位时，电机会突然转动  
**原因**: PWM初始化duty为0，对于反相逻辑意味着全速（Low=ON）  
**解决**: 将`LEDC_DUTY`改为8191（13-bit最大值），对应电机OFF状态

---

### 问题：软启动未触发导致启动电流过大
**现象**: 部分指令显示`startup=11`但无reset过程，速度超调59%  
**原因**: `prev_target_speed[]`数组未正确跟踪速度变化，无法检测0→非零转换  
**解决**: 
1. 添加`prev_target_speed[4]`静态数组跟踪每个电机的目标速度
2. 在PID计算中检测`temp > 0 && prev_target_speed[index] == 0`条件
3. 满足条件时重置`startup_phase=1`和`startup_counter=0`
4. 更新`prev_target_speed[index] = temp`

**验证**: 
- 软启动从startup=1递增至11，历时2秒
- PWM duty从7589平滑降至6553
- 无速度超调现象

---

### 问题：MQTT消息延迟6秒以上
**现象**: 命令发送后6~90秒才收到PCNT_EVENT反馈  
**根因分析**: 
```
14:06:17.569 - 发送 cmd_2_50_5
14:06:39.750 - MQTT断开 (TCP_TRANSPORT_ERROR)
14:06:39~14:07:19 - 8次重连失败
14:07:19.988 - MQTT重连成功
14:07:48.632 - 重新发送 cmd_2_50_5 才得到响应
```
**结论**: 
- 非代码问题：电机控制、PID、PWM逻辑均正常
- 网络环境问题：WiFi信号-70dBm，MQTT连接保持率低
- 6秒延迟实际为91秒延迟，由MQTT重连导致

**建议**: 
1. 改善WiFi信号（目标RSSI > -65dBm）
2. 添加本地命令缓冲，断线时缓存指令
3. 遥测数据使用QoS 0减少网络负载
